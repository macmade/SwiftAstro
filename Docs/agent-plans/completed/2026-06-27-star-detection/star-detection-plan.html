SwiftAstro

Overview

The current MomentStarDetector thresholds the whole image at median + σ·noise, groups connected above-threshold pixels, and keeps blobs by pixel count. On real frames this latches onto noise speckles and nebulosity edges rather than stars: on a deep M42 stack the detections trace the nebula's threshold contour as a green ring, and on M35 thousands of sub-pixel noise clusters are reported as stars (median HFR ≈ 1.25 px). A global threshold fundamentally cannot separate point sources from extended nebulosity or from a near-flat noise floor in a deep stack.

This plan replaces it with a single DAOFIND-style detector: convolve with a zero-sum Gaussian matched filter (no response to smooth backgrounds — kills the nebula ring), find local-maxima peaks above a noise-relative threshold, apply sharpness and roundness cuts (kills noise speckles and elongated artifacts), then measure each survivor with a 2D elliptical Gaussian fit for accurate centroid/FWHM/eccentricity, with HFR measured empirically. The matched-filter scale (expected FWHM) is auto-estimated per image, not configured. Heavy pixel math runs on Accelerate (vImage/vDSP). The MomentStarDetector and the PSFStarDetector stub are removed; the moment math survives as an internal StarMoments helper used for the fit's initial guess and HFR.

Questions & answers

Q: What should the detector optimize for?

A: Purity — detect only confident, well-formed point sources; reject noise, hot pixels, and nebulosity even at the cost of missing faint stars.

Q: May it use Accelerate, or must SwiftAstro stay pure-Swift/cross-platform?

A: Use Accelerate (Apple-only is acceptable for this library's targets).

Q: How far on detection/measurement — matched-filter + moments, + per-star Gaussian fit, or full true PSF fitting + deblending?

A: Don't want to maintain multiple algorithms. Matched-filter + per-star Gaussian fit, provided it improves accuracy and isn't slow. Remove the PSF stub and the existing detector (its math may be kept for tests).

Q: Require an expected FWHM parameter?

A: No — auto-estimate it from the image (bootstrap pass), keeping an optional manual override.

Conventions

  • One type per file; group astro-specific detection code under StarDetection/.
  • Generic, domain-agnostic image primitives belong in the SwiftPixel submodule, not in SwiftAstro — e.g. Gaussian kernels and 2D convolution, which sit alongside SwiftPixel's existing Debayer/Scale/Normalize processors. SwiftAstro keeps only the astronomy-specific usage (mirroring the existing Debayer in SwiftPixel vs. BayerGrayscaleConverter in SwiftAstro split). Modifying the SwiftPixel repo is acceptable when a milestone needs a new generic primitive; such a milestone then yields two commits (SwiftPixel, then the SwiftAstro submodule-pointer bump).
  • Prefer public on library API; reserve internal/private for genuine implementation details.
  • Every added/removed file kept in sync with its package's Xcode project — SwiftAstro.xcodeproj and/or SwiftPixel.xcodeproj — and the SwiftPM target; green under both xcodebuild test and swift test for each affected package.
  • Document all types/members; avoid raw loops; one fix at a time.

Detector pipeline (target)

  1. Validate single-channel, non-empty.
  2. Auto-estimate FWHM: robustly threshold for the brightest obvious stars, size via moments, take the median (skipped if a manual override is set).
  3. Build a truncated zero-sum Gaussian kernel from σ = FWHM / 2.3548 over ~2σ radius.
  4. Convolve (separable Gaussian via vImage, minus a box-mean to realize the zero-sum response).
  5. Robust convolved-noise σc = MAD·1.4826; threshold at thresholdSigma · σc.
  6. Peak detection: strict 8-neighborhood local maxima above threshold; enforce min separation (brighter peak wins).
  7. Sharpness (reject broad blobs / single hot pixels) and roundness (reject elongated) cuts; reject within edge margin.
  8. Measure: moments initial guess → elliptical 2D Gaussian fit (Levenberg–Marquardt); drop non-converged/non-physical fits. FWHM & eccentricity from σmajor/σminor/θ; HFR empirical around the fitted center.
  9. Return StarField.

Default configuration

ParameterDefaultPurpose
expectedFWHMnil (auto)Matched-filter scale; manual override when non-nil
thresholdSigma5.0Detection threshold on the convolved image, in noise σ
sharpnessLow0.2Reject single hot pixels (flux not spread into the neighbours)
minFWHMFactor / maxFWHMFactor0.5 / 2.0Reject fits whose FWHM is far from the matched-filter scale (sub-PSF spikes / broad blobs & nebulosity)
roundnessLimit0.6Reject elongated artifacts, trails (on the fitted eccentricity)
minSeparation= estimated FWHMDeduplicate adjacent peaks
edgeMargin4 pxReject clipped profiles at the border
saturationLevelnilOptional: drop bloomed stars at/above this level

Final values, locked in during Milestone 5. The original plan's sharpnessHigh was dropped: a well-sampled real star's sharpness ≈ 1 (same as a flat blob), so a high-sharpness cut rejected real stars; the minFWHMFactor/maxFWHMFactor band on the fitted size rejects blobs and spikes instead.

Status: Completed
Completed: 2026-06-27
Implementation notes
  • Added a new SwiftAstro/Imaging/ folder with GaussianKernel.swift and Convolution.swift, both public, and the matching test folder SwiftAstroTests/Imaging/. All four files were added to SwiftAstro.xcodeproj (new Imaging groups in both targets) and are picked up automatically by SwiftPM.
  • GaussianKernel(sigma:radiusInSigmas:) samples a normalized isotropic Gaussian over an odd square footprint (size = 2·radius + 1, radius = ⌈radiusInSigmas·σ⌉, clamped to ≥ 1; radiusInSigmas defaults to 2). It exposes values (sum ≈ 1) and zeroSumValues (the same weights with the mean subtracted, sum ≈ 0).
  • Convolution.zeroSumResponse(of:kernel:) produces the matched-filter response. Design choice / deviation from the plan: rather than two literal separable passes (Gaussian blur then a box-mean blur), it runs a single 2D convolution with the zero-sum kernel. By linearity, convolving with (Gaussian − mean) is exactly "Gaussian blur minus box-mean blur" — same result, simpler code — and the kernels are tiny (≈ 7–11 px), so the O(k²) cost is negligible. Heavy math runs on Accelerate via vDSP.convolve (the Double 2D overload, confirmed present in the local SDK Accelerate.swiftinterface).
  • Edge handling is done here, not left to vDSP: the input is padded by replicating its border pixels, convolved, then cropped back to size. This makes the central result independent of vDSP's own border policy and gives a clean ≈ 0 response to a flat field right up to the edges (verified by the flat-field test passing on all pixels including borders).
  • 8 tests, all green under both swift test and xcodebuild test: kernel sums to 1, zero-sum sums to 0, symmetric, centre is the max, odd square footprint; flat field → ≈ 0 everywhere, a delta reproduces the (amplitude-scaled) zero-sum kernel (confirming centering + edge-extend), and a matched Gaussian blob peaks exactly at its centre.
  • Post-review relocation (see Review comments): both files and their tests were moved out of SwiftAstro into the SwiftPixel submodule as generic primitives — SwiftPixel/SwiftPixel/GaussianKernel.swift and Convolution.swift, with tests SwiftPixelTests/GaussianKernel.swift and Convolution.swift (renamed to SwiftPixel's Test_<Type> convention and rewritten to build fixtures inline rather than via SwiftAstro's SyntheticStarField). Docs were generalized (no more "star detector" wording). The new SwiftAstro/Imaging/ folder was removed; SwiftAstro.xcodeproj reverted. Files added to SwiftPixel.xcodeproj. Both packages green under swift test and xcodebuild test. The M4 detector will import SwiftPixel and call Convolution.zeroSumResponse.
Review comments

  • Unit tests: recover known synthetic Gaussian params (center, σ, θ, bg) within tolerance under noise; report non-convergence on garbage.

Status: Completed
Completed: 2026-06-27
Implementation notes
  • Added SwiftAstro/StarDetection/GaussianFit.swift: a public enum namespace with a nested Parameters struct (amplitude, centre x/y, σx, σy, θ, background) and fit(samples:initialGuess:) -> Parameters?. Parameters exposes the model evaluation value(atX:y:) and a Parameters(moments:amplitude:background:) initializer that seeds the guess from a StarMoments measurement (centre from the centroid; σ-major/σ-minor and θ from the second-moment ellipse's eigen-decomposition).
  • Design choice — numerical Jacobian: the LM solver uses a central-difference Jacobian rather than hand-coded analytic derivatives. The 7-parameter rotated model makes analytic derivatives error-prone, while finite differences over the small window are cheap and robust. Flag this in case you'd prefer analytic derivatives for speed.
  • Levenberg–Marquardt with Marquardt diagonal damping (scales JᵀJ's diagonal by 1 + λ; λ shrinks on accepted steps, grows on rejected ones), a small Gaussian-elimination solver with partial pivoting for the 7×7 normal equations (index loops kept there as the clearest form for fixed-size elimination — flagging in case you want it replaced with an Accelerate/LAPACK solve), convergence on relative cost improvement < 1e-8, and capped iterations.
  • Reject / quality guards: fit returns nil when the result is non-physical (amplitude not finite/positive, σ not finite/positive, σ larger than the window span, non-finite centre/background) or no better than a flat background (fit residual ≥ the constant-mean residual). σ signs are normalized to positive on return.
  • Added GaussianFitTests: recovers a clean round Gaussian from a rough guess; recovers a rotated elliptical Gaussian under noise when seeded from StarMoments (checks sorted σ, centre, amplitude and RMS fit residual, which validates orientation); rejects a central dip (negative peak) and a flat field; and checks the model peaks at its centre. Deterministic noise reuses the test helper's DeterministicNoise.
  • Both files added to SwiftAstro.xcodeproj. Green under both swift test (40 tests) and xcodebuild test.
  • Post-review relocation (see Review comments): the generic fitter moved to the SwiftPixel submodule — SwiftPixel/SwiftPixel/GaussianFit.swift (model + fit + LM solver) with generic tests SwiftPixelTests/GaussianFit.swift (round recovery, dip/flat rejection, model peak). The one astro coupling, the Parameters(moments:amplitude:background:) seeding initializer, stays in SwiftAstro as an extension in SwiftAstro/StarDetection/GaussianFit+StarMoments.swift, with its end-to-end "seed from moments and fit" test in GaussianFitStarMomentsTests. Docs were generalized (blob-like features, not stars/PSFs). Both packages green under swift test and xcodebuild test.
Review comments
  • The Gaussian fitter's core (model + Levenberg–Marquardt + solver) is generic curve-fitting, so it belongs in SwiftPixel like the other generic primitives. Only the StarMoments-seeding initializer is astro-specific (and cannot move, since SwiftPixel must not depend on SwiftAstro); it stays in SwiftAstro as an extension on the SwiftPixel type.

Implementation notes
  • Added SwiftAstro/StarDetection/MatchedFilterStarDetector.swift, a public struct conforming to StarDetecting, with a nested Configuration carrying the plan's defaults (expectedFWHM nil/auto, thresholdSigma 5, sharpnessLow/High 0.2/1.0, roundnessLimit 0.5, minSeparation nil → estimated FWHM, edgeMargin 4, saturationLevel nil).
  • Pipeline: median background → FWHM (override or auto-estimate) → GaussianKernel at σ = FWHM/2.3548 → Convolution.zeroSumResponse → robust convolved-noise σc (MAD·1.4826) and threshold at thresholdSigma·σc → strict 8-neighbour local maxima → min-separation dedup (spatial-hash grid, brighter wins) → per-peak edge/saturation/sharpness/roundness cuts → moment-seeded GaussianFit → Star. Returns an empty field when σc is not positive (no measurable noise).
  • FWHM auto-estimation (static estimateFWHM(in:), internal so it is unit-tested directly): thresholds at background + 10·noise to isolate the brightest stars, sizes each one's body (pixels > bg + 3·noise) over a ±10 px window via StarMoments.fwhm, and takes the median (robust to non-stellar peaks). Falls back to 3.0 px if nothing is sizeable.
  • Metric definitions (DAOFIND-inspired, to be locked in M5): sharpness = mean of the 8 neighbours' background-subtracted values ÷ the peak's — ≈ 0 for a single hot pixel, ≈ 1 for a flat blob, mid-range for a real star; roundness = the moment eccentricity (pre-fit cut). Final star FWHM = 2.3548·√(σmajor·σminor) and eccentricity = √(1 − σminor²/σmajor²) from the fit; HFR is measured empirically about the fitted centre via the new StarMoments.halfFluxRadius(samples:background:aroundX:y:) (added this milestone, purely additive); flux is the integrated Gaussian 2π·A·σx·σy.
  • MatchedFilterStarDetectorTests (8): detects 3 planted stars at their positions; zero on a smooth gradient (new SyntheticStarField.addingGradient helper) and on pure noise; rejects hot pixels; recovers a star's FWHM/eccentricity; auto-FWHM ≈ planted; honours a manual override; rejects multi-channel input.
  • Files added to SwiftAstro.xcodeproj. Green under both swift test (44 tests) and xcodebuild test. (MomentStarDetector still present and tested; it is removed in M6.)

Status: Completed
Completed: 2026-06-28
Implementation notes

Validation on the real frames exposed problems that pure parameter tuning could not fix, so (with sign-off) M5 reworked parts of the detector — the agreed "Option 1": fix the real failures with simple, self-calibrating cuts rather than reproducing DAOFIND's exact formulas.

  • Root cause of the bad real-frame numbers: a global background. The per-peak measurement and the FWHM bootstrap both subtracted the global median, but the real sky varies (gradient/vignetting). On H3.69 the global median sat below the local sky, so the bootstrap body filled the window (estimated FWHM ballooned to 14–84 px) and HFR inflated (median ≈ 16 px vs. a true ≈ 3.7). Fix: estimate a local background (median of the measurement window) for the moments, the fit seed, and — via the fitted background — the HFR; size each bootstrap star over its body against its own local background.
  • FWHM bootstrap reworked to size each bright star by the empirical half-flux radius of its body (FWHM = 2·HFR), which is core-dominated and robust to the broad wings of bright/near-saturated stars that biased the second-moment version high.
  • Purity cuts redesigned. The old sharpness ≤ sharpnessHigh cut was degenerate — a well-sampled real star's neighbour/centre ratio ≈ 1, same as a flat blob — and rejected genuine stars (H3.69 collapsed to 1 detection). Now: keep the low-sharpness cut (rejects hot pixels) pre-fit; after the fit, require the fitted FWHM within a factor of the estimated FWHM (rejects sub-PSF spikes and over-large blobs/nebulosity) plus the roundness cut on the fitted eccentricity. Configuration changed: removed sharpnessHigh; added minFWHMFactor (0.5) and maxFWHMFactor (2.0); roundnessLimit 0.5 → 0.6.
  • Performance + a second relocation to SwiftPixel. Statistics.median was the hot spot (a generic comparison sort, seconds per call on multi-megapixel frames, several calls per detection). It was first given an Accelerate fast path, then — on review — recognized as a generic primitive duplicating SwiftPixel's existing vDSP-backed PixelUtilities.percentileBounds. So median + MAD were folded into SwiftPixel.PixelUtilities, right next to percentileBounds (the Double median reuses it — a median is the 50th percentile — for a single sort/percentile implementation). They were folded into PixelUtilities rather than a new Statistics type to avoid a name clash with the existing HistogramStatistics. SwiftAstro's Core/Statistics.swift was deleted and its callers now use PixelUtilities via import SwiftPixel. The redundant global-background pass was also dropped from detectStars.
  • Final real-frame results (release): H3.69 → 52 stars, median HFR 3.75 (filename reports 3.69), median FWHM 7.69; M35 → 405 stars, median HFR 2.06, FWHM 4.28; M42 → 157 stars (vs. the old detector's ~1031 nebula-ring), median HFR 2.01. Chosen defaults (locked in): thresholdSigma 5, sharpnessLow 0.2, roundnessLimit 0.6, minFWHMFactor 0.5, maxFWHMFactor 2.0, edgeMargin 4, auto FWHM, minSeparation = estimated FWHM.
  • Timing — honest note: ~2.6–3.4 s per full frame in release, not the hoped-for sub-second. The cost is the 8-megapixel matched-filter convolution plus a Levenberg–Marquardt fit per star; the median fix removed the worst offender (tens of seconds). Sub-second would need further work (downsampled bootstrap, analytic Jacobian, or a coarser convolution) and is left as a possible follow-up, not forced into M5.
  • Tests: RealFrameValidationTests retargeted to MatchedFilterStarDetector via FITSImageDecoder.detectionImage. H3.69 runs on the full frame (headline median-HFR ≈ 3.69 ±20%). Since every star is Gaussian-fitted, the full multi-megapixel frames are far too slow in an unoptimized (Debug) test build — and xcodebuild cannot test in Release (testability is off) — so the M35 and M42 checks run over a 512×512 central crop (real data, bounded work): M35 asserts a plausible count and a stellar median HFR (> 1.5 px); M42 asserts a bounded count (no nebula contour) and a stellar HFR. The M42 frame loads from the SwiftFITS submodule by #filePath path (no copy, no bundling), per the FITScope convention. New test helpers: detection-image loaders, the M42 loader, a crop, and SyntheticStarField.addingGradient.
  • Deviation from the plan wording for M42: the plan asked for "the central nebula region essentially clear of detections," but M42's centre is the star-rich Trapezium, so a handful of real detections there is correct. The anti-nebulosity property is instead asserted as a bounded count over the bright core (vs. the old ring of hundreds).
  • The throwaway diagnostic was removed (file + project entries). Full suites green under both xcodebuild test and swift test for SwiftAstro and SwiftPixel; MomentStarDetector still present (removed in M6).
Review comments
